# 場所アーキタイプ × 困り（needType）— 実装計画書（Phase 10）

> **目的**: 不満投稿の **空間文脈** を、島上の **場所セット（都市原型）** ＋ **オーブ（個別の声）** で表現する。③（状況が見えにくい）→ ④（受容）への架け橋を具体化する。  
> **前提**: Phase 8（バリア判定・プラン選択）完了、Phase 9（水辺・フェリー）稼働、`docs/27`（設計根幹）、`docs/26`（ARループ閉じ）と連携可。  
> **やらないこと**: 場所×困りの全組み合わせ専用マップ、PLATEAU全域取り込み、Unity移行、実世界GPS一致。

---

## 0. エグゼクティブサマリー

### なぜ Phase 10 か

| 現状 | ギャップ |
|------|----------|
| 不満は `bugs[]` のオーブ＋オーバーレイで表示 | **「駅の段差」など場所の型**が空間に出ない |
| 島拡張は汎用チャンク＋ランダム不満 | 投稿内容と**無関係な地形**が増える |
| AR投稿は needType のみ構造化 | **どの場所型に載せるか**のルールがない |

### 設計の3原則

1. **セット数 ＝ 場所型の数**（最大8）。困りごとに別マップは作らない。  
2. **セットは器、オーブは声**。セットがなくてもオーブ＋写真で必ず受けられる。  
3. **needType はセット内の演出**（段差・暗所・看板なし等）を切り替えるだけ。

### 工数目安

| 段階 | 内容 | 目安 |
|------|------|------|
| **10-A** | データ・ルーティング・スポーン基盤 | 3〜4日 |
| **10-B** | 場所セット MVP 4種（①②③④） | 4〜6日 |
| **10-C** | AR投稿連携・自動配置（`docs/26` と統合） | 2〜3日 |
| **10-D** | 残り場所型・演出拡張 | 随時 |

**推奨PR**: 4本（基盤 / MVPセット / 投稿連携 / 残り場所）

## 1. 概念モデル

### 1.1 用語

| 用語 | 意味 |
|------|------|
| **場所型（placeArchetype）** | 駅・道路・公園など、島に出す **空間セットの種類**（1型＝1プリセット） |
| **困り（needType）** | ARで選ぶ P/V/I/M/R/L/S/C。セットの **演出バリアント** を決める |
| **サイト（placeSite）** | 島上に配置された場所セットの **1インスタンス**（チャンク群＋プリセットブロック＋アンカー） |
| **アンカー（bugAnchor）** | セット内でオーブを置く **決まった座標**（相対オフセット） |
| **演出（accent）** | needType に応じたライト・段差・看板の有無など（プリセットの差分） |

### 1.2 プレイヤー体験（目標）

```
AR投稿（困り＋写真＋一言）
  → システムが場所型を決定
  → 島に「駅まわり」等のセットが出現（初回）または既存サイトに追加（2件目以降）
  → セット内のアンカーにオーブ（投稿紐づけ）
  → 歩いて発見 → 既存 Phase 8 フロー（プラン選択 → DIY → 解決）
```

### 1.3 フォールバック

| 条件 | 動作 |
|------|------|
| 場所型プリセット **あり** | サイト出現 ＋ アンカーにオーブ |
| 場所型 **未実装**（例：⑦商業） | 汎用チャンク ＋ オーブのみ（現状同等） |
| 演出 **未定義** | セット骨格のみ ＋ オーブ主（写真・コメント） |

## 3. needType → 場所型ルーティング（投稿配置先候補）

**セットを増やす表ではない。** 投稿の場所型が未指定のときの **デフォルト候補順**。

| needType | 候補場所型（優先順） |
|----------|----------------------|
| P 歩きにくい | transit → plaza → road → lane |
| V 見えにくい | lane → park → waterfront |
| I 迷いやすい | transit → plaza → road → commerce |
| M 汚れ・荒れ | park → plaza → waterfront → commerce |
| R 休みにくい | plaza → park → waterfront → commerce |
| S 不安 | lane → park → road → waterfront |
| L 行き来しにくい | transit → waterfront → road |
| C 頼れる場がない | lane → park → plaza → campus |

### 3.1 場所型の決定ロジック（優先順）

1. 投稿に **`placeArchetype` 明示**（将来 AR Step 任意選択）  
2. **`mapPin` + 柏の葉エリア分類**（将来。MVP はスキップ）  
3. コメント **キーワード**（駅・バス・公園・学校…）  
4. 上表の **needType デフォルト先頭**  
5. フォールバック: `road`

## 4. 場所型 × needType — セット内演出（アクセント表）

各場所型プリセットは **骨格1つ**。セルは **その needType で強調する演出**（未記載はオーブ主）。

### 4.1 `transit`（駅・バス）

| needType | セット演出（examples） | オーブ |
|----------|------------------------|--------|
| P | ホーム縁 `half` 段差、スロープ未設置 | アンカー上 |
| I | 分岐路、看板なし | 同上 |
| L | 長い階段／乗り場までの段差 | 同上 |
| V/S | 照明弱め（`worldTime` 夜固定は任意） | **主** |

### 4.2 `plaza`（広場・駅前）

| needType | セット演出 | オーブ |
|----------|------------|--------|
| P | 広場縁の段差、イベント障壁比喩 | アンカー |
| R | ベンチなし（`bench` 意図的配置なし） | 同上 |
| M | ゴミ比喩ブロック／色汚れ | 同上 |
| S | 開けすぎた広場・灯少なめ | 主 |

### 4.3 `road`（道路）

| needType | セット演出 | オーブ |
|----------|------------|--------|
| P | 横断歩道前 `half`、歩道断線 | アンカー |
| V | 街灯間隔広い | 同上 |
| S | 車道隣接・歩道狭い | 主 |

### 4.4 `lane`（街区）

| needType | セット演出 | オーブ |
|----------|------------|--------|
| V | 路地・照明1灯のみ | アンカー |
| S | 死角、建物に挟まれた道 | 主 |
| P | 細道に段差 | アンカー |
| C | ベンチ・見守りなし | 主 |

### 4.5 その他（Phase 10-D）

- `campus`: P 横断、V 通学路暗所  
- `park`: R ベンチ不足、M 荒れ  
- `waterfront`: L フェリー未接続（既存 `transit_link`）  
- `commerce`: I 看板混乱、R 待ち場所なし

## 5. データ構造

### 5.1 定数 — 新規 `src/constants/placeArchetypeData.js`

```typescript
// 概念型

type PlaceArchetypeId =
  | 'transit' | 'plaza' | 'road' | 'lane'
  | 'campus' | 'park' | 'waterfront' | 'commerce';

type PlaceAccentId =
  | 'steps' | 'dark' | 'no_sign' | 'no_bench'
  | 'narrow_path' | 'broken_sidewalk' | 'none';

type PlacePresetBlock = {
  shape: string;
  material: string;
  pos: [number, number, number];  // サイト原点相対
  scale?: [number, number, number];
  rotation?: number;
};

type PlaceArchetypePreset = {
  id: PlaceArchetypeId;
  label: string;
  kashiwanohaLabel?: string;
  chunkLayout: 'single' | 'ring_fragment';  // MVP は single 10x10
  defaultAccentByNeed: Partial<Record<NeedType, PlaceAccentId>>;
  accents: Record<PlaceAccentId, {
    blocks?: PlacePresetBlock[];
    lighting?: 'normal' | 'dim';
    bugAnchor: [number, number, number];
  }>;
  fallbackAccent: PlaceAccentId;  // 通常 'none'
};
```

### 5.2 島チャンク拡張 — `islandChunks[]`

```json
{
  "id": "site_transit_001",
  "pos": [20, -0.3, 0],
  "size": [10, 0.6, 10],
  "kind": "place_site",
  "placeArchetype": "transit",
  "dropIn": true
}
```

- `kind: 'place_site'` … 汎用 `main` / `remote` と並ぶ第三種（描画・スポーン判定用）

### 5.3 プリセットブロック — `placedBlocks[]` 拡張

```json
{
  "id": "preset_xxx",
  "shape": "half",
  "material": "stone",
  "pos": [22, 0.5, 1],
  "presetSource": "place_site:site_transit_001",
  "presetLocked": true
}
```

- `presetLocked: true` … プレイヤー削除不可（MVP）。解決後に一部解放は 10-D。

### 5.4 不満・クエスト拡張

```json
{
  "placeArchetype": "transit",
  "placeSiteId": "site_transit_001",
  "sourceQuestId": 102,
  "accentApplied": "steps"
}
```

`docs/26` の `sourceQuestId` と **同一フィールド**で連携。

### 5.5 サイト台帳 — 新規 `placeSites[]`（store、任意）

チャンク＋アンカー＋投稿数をまとめて管理。MVP は `islandChunks` の `placeArchetype` だけでも可。  
複数投稿を同一サイトに載せるときに `placeSites[]` があると楽。

## 6. 処理フロー

### 6.1 `resolvePlaceArchetype(input)` — `src/utils/placeRouting.js`

**入力**: `{ needType, comment, placeArchetype?, mapPin? }`  
**出力**: `PlaceArchetypeId`

### 6.2 `resolveAccent(archetype, needType)` — 同上

**出力**: `PlaceAccentId`（`defaultAccentByNeed[needType]` ?? `fallbackAccent`）

### 6.3 `spawnPlaceSite({ archetype, accent, worldAnchorPos? })` — `src/store/helpers/placeSiteFactory.js`

1. 空きリング位置を計算（既存 `createRingExpansionChunks` 流用）  
2. `kind: 'place_site'` チャンク追加  
3. `buildPresetBlocks(preset, accent, siteOrigin)` → `placedBlocks` に追加  
4. `bugAnchor` 世界座標を返す  

### 6.4 `spawnBugFromQuest(quest)` — `questToBug` 拡張（`docs/26` 統合）

1. `resolvePlaceArchetype(quest)`  
2. 既存 `placeSiteId` があるか（同一 archetype で近接）→ あれば再利用  
3. なければ `spawnPlaceSite`  
4. `normalizeBug({ ...quest, pos: anchor, placeArchetype, placeSiteId, sourceQuestId })`  
5. `bugs.push`、トースト「〇〇（駅まわり）に載せました」

### 6.5 既存シード不満

`createDefaultQuests` / `createDefaultBugs` に `placeArchetype` を付与。  
デモ島ロード時に **シード用サイト1件**（`transit`）を事前配置してもよい。

## 13. 卒論・限界の書き方（1段落）

> 本実装は柏の葉の実景の複製ではなく、帰納的に抽出した都市空間原型を島上に配置する方式である。場所型の数は実装段階に応じて段階的に増やすが、困りの型ごとに独立した空間モデルを用意するのではなく、同一原型に対する演出差分とオーブ表示により文脈を補う。PLATEAU等による形状取り込みは将来の精度向上として位置づける。
