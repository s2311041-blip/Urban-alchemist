# バリアに関する設計の根幹

> **この文書の役割：** Urban Alchemist の「シリアス」部分——バリアをどう捉え、何のために体験させるか——の本質だけを書く。  
> **書かないこと：** コード・データ構造・実装状況・ファイル名。

---

## 1. 何のためのゲームか

本ゲームのバリア関連は、**娯楽としての街づくり**ではなく、まちづくり参加における **RQ2（受容）** を担う。

| | 内容 |
|---|------|
| **問い** | ワークショップ参加者は、ライト層が投稿した困りごとを、どう**理解し受容**できるか |
| **手段** | 他者の声をゲーム空間に載せ、**歩いて見つけ・読み・手を動かして改善を試す**体験（視点置換） |
| **狙わないこと** | 障害の完全再現、UDの網羅講義、合意形成の達成宣言、ゲームが楽いから集まる設計 |

AR投稿（RQ1）は **「誰の・どこで・どんな困り」という素材を集める入口**。ゲームは **「集めた素材を、議論の場の人が自分ごととして咀嚼する場」**。

---

## 2. バリアをどう捉えるか

### 2.1 人ではなく「状況」として見る

- バリアは特定の個人の欠けではなく、**環境・運用・つながり**として起きる。
- 常に **「誰にとって」** の問題かを意識する（高齢者、車いす、ベビーカー、子ども、観光客、みんな…）。
- **いつ**（朝・夜・常時）・**どの程度**かも、同じ場所でも変わる。

### 2.2 困りの広がり（3つのスケール）

| スケール | 捉え方 |
|----------|--------|
| **一点** | この場所だけの問題（暗い角、一か所の段差） |
| **動線** | 通り道・ルート沿いの連続した問題（段差が続く道、案内が途切れる道） |
| **周辺** | エリア全体の偏り（休める場所が少ない、頼れる場所がない） |

### 2.3 要因の種類（3つの軸）

| 要因 | 内容 |
|------|------|
| **物理** | 段差、暗さ、狭さなど、身体や移動に直接効くもの |
| **ソフト** | 案内不足、汚れ・荒れ、維持の行き届かなさなど |
| **人的** | 孤立、不安、見守りや支援につながりにくさなど |

### 2.4 困り方のジャンル（投稿者・プレイヤー向けの8類型）

専門用語ではなく、**「どんな困り方か」** で分ける。

| ジャンル | 例 |
|----------|-----|
| 歩きにくい | 段差、狭さ、通りづらさ |
| 見えにくい | 暗さ、死角 |
| 迷いやすい | 案内不足 |
| 汚れ・荒れ | 清掃・維持不足 |
| 休みにくい | 座る・待つ場所が足りない |
| こわい・不安 | 心理的に安心できない |
| 行き来しにくい | アクセス・移動手段の不足 |
| 頼れる場がない | 支援・見守りにつながりにくい |

## 3. プレイヤーに何を体験させるか

```
他者の投稿（写真・一言・誰にとって）に出会う
  → その状況を読む（哀れむのではなく、文脈として理解する）
  → 改善の「型」を選ぶ（一つに決まらない）
  → 手を動かして街に手を入れる
  → 変化を見て、まだ残る課題にも気づく（完全解決ではないことも含む）
```

**視点置換**とは、他者の属性やコメントを通じて、「自分とは違う状況で困っている人がいる」と**対等な市民として**向き合うこと。  
マイノリティ福祉主義的な **哀れみ（pity）** ではなく、**共感的理解と受容**を目指す。

---

## 4. 解決の考え方——「正解一つ」にしない

従来の話し合い中心の参加では、**相手の感情や状況が見えにくい**。  
本ゲームでは、集めた声を **空間・写真・属性** 付きで見せたうえで、改善も **複数の型** から選ばせる。

| 改善の型（例） | 考え方 |
|----------------|--------|
| 物理整備 | 段差解消、照明など |
| 迂回・動線 | 直せないときの別ルート |
| 案内・情報 | 迷いを減らす |
| 維持・使いやすさ | 汚れ・休み場所など |
| 見守り・つながり | 人的な不安への手当て |
| 交通・アクセス | 行き来そのものの接続 |

同じ不満でも、**状況によって有効な手は違う**。  
「建築＝唯一の正解」ではなく、**トレードオフ**（例：通行は良くなったが、勾配が急だとまだ負担が残る）も示す。

## 5. ARとゲームの役割分担

| 段階 | 役割 |
|------|------|
| **AR（RQ1）** | ライト層が、現地・地図から、**低負荷で**困りを記録する（空間注釈） |
| **ゲーム（RQ2）** | WS層が、その記録を**体験空間で受け取り**、理解・受容・改善試行につなげる（視点置換） |

一本の流れ：

> **集める（Consult）→ 載せる → 体験して受容する（Involve）**

ARとゲームは競合する道具ではなく、**プロセス上の前後関係**にある。

---

## 6. 設計で守っていること

1. **投稿は軽く、理解は厚く** — 写真と一言で済ませ、構造化は体験側で支える  
2. **誰にとってかを常に見える化する** — バリアを抽象化しすぎない  
3. **複数プラン・複数正解** — 単一の「バリアフリー完了」にしない  
4. **残る課題も正直に** — 改善しても全部は直らない、と伝える  
5. **娯楽化の主役にしない** — ポイントや島拡張は手応えの補助。主役は他者理解  

---

## 7. 専門家・論文向けの一行

> 他者の都市バリアを、**誰にとって・いつ・どの広がりで**起きているかとして提示し、**一つの正解にしない複数の改善**を試すことで、まちづくり参加者の**対等な受容**を促す参加型シリアスゲームである。

---

## 8. 意図的にやらないこと

- 障害体験の「なりきり」や完全シミュレーション  
- 当事者の物語の消費・インスピレーション化  
- UDチェックリストの網羅教育  
- 「この配置でバリアフリー達成」の単純ゲーム化  
- ARを頑張れば参加が増える、という主張（参加はループ設計・受容体験まで含めて考える）

---

*根幹の一行：バリアは「直す対象」ではなく、他者の状況として読み、複数の手で向き合う——その体験が、本ゲームのシリアス部分の本質である。*
