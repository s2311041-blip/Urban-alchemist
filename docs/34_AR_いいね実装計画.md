# AR 投稿 — いいね（pt 消費）実装計画

> **目的:** pt に **使い道（sink）** を与え、投稿以外の **軽い参加**（共感）を可能にする。  
> **関連:** [33_AR_投稿設問の吟味.md](./33_AR_投稿設問の吟味.md) §9、[25_AR投稿_参加要因_先行研究レビュー.md](./25_AR投稿_参加要因_先行研究レビュー.md)

## 0. エグゼクティブサマリー

| 項目 | 内容 |
|------|------|
| 新機能 | 他者投稿への **いいね**（1 pt 消費 / 1 回）＋ **共感ランキング**画面 |
| 前提 | **みんなの共感を見る**には Supabase クラウド同期が必須 |
| 工数目安 | **2〜3 日**（PR 2 本に分割可） |
| やらないこと | 件数の個別強調表示、投稿者への pt 還元、コイン連携 |

## 1. ルール（確定案）

| ルール | 内容 |
|--------|------|
| コスト | **1 いいね = 1 pt 消費** |
| 対象 | **他人の投稿のみ**（`authorId !== mine`） |
| 重複 | 1 ユーザー × 1 投稿 = **1 回まで**（取り消し可とするか → **v1 は不可・追加のみ**） |
| 残高不足 | ボタン disabled ＋「pt が足りません（投稿で貯まります）」 |
| 投稿者報酬 | **なし**（インフレ・競争回避） |
| 個別カード | **件数は出さない**（♡ 済みかどうかのみ） |
| 一覧 | **共感ランキング**で順位として相対表示（下記 §5.3） |

### 1.1 pt 残高の考え方

| 概念 | 定義 |
|------|------|
| **獲得 pt** | 自分の投稿の `pointsAwarded` の合計 |
| **消費 pt** | 自分が付けたいいね数 × 1 |
| **利用可能 pt** | 獲得 − 消費 |

クラウドモードでは **Supabase 上の ar_likes 件数** から消費を再計算できる（localStorage 改ざんに強い）。

ローカルモードでは Zustand persist の `likeRecords[]` で代用。

## 2. データモデル

### 2.1 新テーブル `ar_likes`

```sql
create table public.ar_likes (
  annotation_id text not null references public.ar_annotations (id) on delete cascade,
  user_id uuid not null references auth.users (id) on delete cascade,
  created_at timestamptz not null default now(),
  primary key (annotation_id, user_id)
);

create index ar_likes_user_idx on public.ar_likes (user_id);
create index ar_likes_annotation_idx on public.ar_likes (annotation_id);

alter table public.ar_likes enable row level security;

-- 全員が件数集計のため read 可
create policy "ar_likes_read_all" on public.ar_likes for select using (true);

-- 自分の user_id のみ insert
create policy "ar_likes_insert_own" on public.ar_likes for insert
  with check (auth.uid() = user_id);

-- v1: delete 不可（取り消しなし）
```

ファイル: `supabase/migrations/003_ar_likes.sql`

### 2.2 アノテーションへの付与

fetch 時に **likeCount** と **likedByMe** を JOIN または 2 クエリで付与:

```javascript
{
  ...annotation,
  likeCount: 3,
  likedByMe: true,
}
```

payload JSONB には書かない（正規化のため別テーブル）。

### 2.3 レート制限（任意・v1 省略可）

1 時間 30 いいね等。スパム対策が必要になったら DB トリガーで追加。

## 3. API 層

### 3.1 新規ファイル

| ファイル | 役割 |
|----------|------|
| `src/ar/api/likesSupabase.js` | `fetchLikeSummary`, `insertLike`, `fetchUserLikeIds` |
| `src/ar/api/likesClient.js` | Supabase / ローカル分岐 |

### 3.2 関数

```javascript
// 同期時: annotation id 一覧 → { counts: Map, myIds: Set }
fetchLikeSummary({ annotationIds, userId })

// いいね実行（残高チェック → insert）
likeAnnotation(annotationId) → { likeCount, pointsAvailable }

// 利用可能 pt
fetchPointsBalance(userId) → { earned, spent, available }
```

**残高チェック（サーバー側）:** v1 は insert 前に RPC または Edge Function なし。**クライアント + RLS** で足りる（ワークショップ規模）。  
v2 で `like_annotation()` RPC（earned − count(likes) ≥ 1 を DB 内検証）を推奨。

### 3.3 ローカルモード

`localStorage` キー `urban_alchemist_ar_likes_v1`:

```json
{ "likes": [{ "annotationId": "ar-...", "at": 1710000000 }] }
```

## 4. Store（`useArPostingStore`）

### 4.1 追加 state

| フィールド | 用途 |
|------------|------|
| `likeSummary` | `{ [annotationId]: { count, likedByMe } }` |
| `pointsAvailable` | 表示用残高（sync 時再計算） |

`totalPoints`（累計獲得）は **そのまま維持**。UI に「利用可能 pt」を追加表示。

### 4.2 アクション

```javascript
syncAnnotations()     // 既存 + fetchLikeSummary をマージ
likeAnnotation(id)    // pt 不足なら throw、成功で summary 更新
refreshPointsBalance()
```

### 4.3 persist

`likeSummary` は **persist しない**（sync で再取得）。  
ローカルモードのみ `localLikes` を persist。

## 5. UI

### 5.1 表示方針 — **件数よりランキング**

個々のカードに「♡ 3」と出すより、**順位付き一覧**の方が「みんながどこに共感しているか」が **相対的** に伝わる。

| 場所 | 表示 |
|------|------|
| ピンカード・図鑑リスト | ♡ ボタンのみ（**件数非表示**）。押下済みは塗りつぶし |
| **共感ランキング**（新画面） | likeCount 降順の **1位・2位…** 。写真サムネ + 一言 + needType タグ |
| ホーム | 「共感ランキングを見る」入口（記録図鑑と並列でも可） |

ランキング画面では **順位番号**（🥇🥈🥉 / 1, 2, 3…）は出すが、**生の件数（「15いいね」）は v1 では出さない**（順位だけで相対比較）。  
分析・運用で必要なら JSON エクスポートや Supabase Dashboard で件数確認。

### 5.2 共感ランキング画面（`ArEmpathyRanking.jsx`）

```
┌─────────────────────────────────────┐
│ ← ホーム    共感ランキング           │
├─────────────────────────────────────┤
│ みんなが共感した記録（上位10件）      │
├─────────────────────────────────────┤
│ 1  [thumb] 段差が高くて…  🟦歩きにくい│
│ 2  [thumb] 夜は暗くて…   🟨見えにくい│
│ 3  [thumb] ベンチが…     🟩休みにくい│
│ …                                   │
│ （同数の場合は新しい投稿を上）        │
└─────────────────────────────────────┘
```

- データ: `annotations` + `likeSummary` を **likeCount 降順** でソート（0 件は末尾 or 非表示）
- タップ → 既存 `ArPinCard`
- スコープ: 江東区同期分すべて（図鑑「みんな」と同じプール）

### 5.3 配置

| 画面 | 要素 |
|------|------|
| **ArPinCard** | ♡ のみ（件数なし）。自分の投稿は非表示 |
| **ArFieldGuide** | リスト行に ♡。上部タブに **「ランキング」** 追加（またはホームから遷移） |
| **ArHomeScreen** | 「共感ランキング」カード |
| **ArHomeScreen** | 「利用可能 pt」（任意） |

### 5.4 インタラクション

```

ランキング → sync 後に再ソート（いいね直後は楽観更新）
```

## 7. Supabase は必須か

| 機能 | Supabase なし | Supabase あり |
|------|---------------|---------------|
| いいねを **自分の端末だけ** 記録 | ○（localStorage） | ○ |
| **複数人の共感** を集計 | × | ○ |
| **ランキング**（みんなの相対） | ×（意味がない） | ○ |
| pt 残高の改ざん耐性 | △ 弱い | ○ |

**結論:** 共感ランキングまで含めるなら **Supabase 設定が前提**。UI 実装はローカルで先行可能だが、**結合テスト・本番は Supabase 後**。
