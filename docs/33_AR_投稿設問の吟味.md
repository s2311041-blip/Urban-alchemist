# AR 投稿 — 設問項目の吟味

> **目的:** RQ1（ライト層の空間注釈）として、今の設問が **摩擦（列 B）** と **文脈記録（列 C）** のバランスに合っているか見直す。  
> **関連:** [27_バリア_シリアスゲーム機能まとめ.md](./27_バリア_シリアスゲーム機能まとめ.md)、[19_表2_意見収集手法の目的適合比較.md](./19_表2_意見収集手法の目的適合比較.md)

## 1. 現状の設問一覧

### 1.1 共通（撮影・位置 — 設問ではないが必須工程）

| 項目 | 取得方法 | RQ1 での役割 |
|------|----------|--------------|
| 写真 | カメラ撮影 | 列 C：状況の記録 |
| 位置（GPS + 方角） | 自動 | 空間注釈の核心 |

### 1.2 困りごと（Bad）— 4 ステップ

| 2 | 困りごと（needType） | 8 | ○ |

実装: `src/ar/components/ArPostForm.jsx`、`src/constants/barrierData.js`

### 1.3 良い場所（Good）— 3 ステップ

| ステップ | 項目 | 必須 |
|----------|------|------|

| 3 | 自由記述 | ○ 1 字以上 |

## 2. 各項目の「なぜ聞くか」

| 項目 | RQ1（集める） | RQ2（受容） | 優先度 |
|------|---------------|-------------|--------|
| needType | 困りの型をざっくり分類 | ゲーム内プラン候補・オーブ種別 | **高** |
| placeArchetype | 江東区のどんな場か | 島上の場所型チャンク（将来） | **高** |
| affectedGroups | 「誰にとって」 | 視点置換の入口 | **高** |
| comment | 写真だけでは足りない文脈 | プレイヤーが読む物語 | **最高** |
| timeTag | 同地点でも夜だけ等 | 状況理解 | 中 |
| severity | 優先度のヒント | クエストの重み付け | 中〜低 |
| 写真 | 列 C の本丸 | 視覚的理解 | **最高** |
| GPS | 空間注釈 | 地図・AR 表示 | **最高** |

**表2の列 B（参加の重さ）** から見ると、Bad 投稿は **タップ数が多い**（8+8+6+5+3+テキスト）。ライト層向けには削る余地がある。

## 3. 困りごと 8 類型 — 重なりの整理

> **ステータス:** needType の **コード変更は保留**。UI は [35_AR_困りごとUI_PPS四色と設問改訂.md](./35_AR_困りごとUI_PPS四色と設問改訂.md)（PPS 4 色グループ）へ移行。

現行ラベル（`NEED_CATEGORY_OPTIONS`）:

| needType | ラベル | hint | ゲーム type への写像 |
|----------|--------|------|----------------------|
| P | 歩きにくい | 段差・狭さ | danger |
| V | 見えにくい | 暗さ・死角 | dark |
| I | 迷いやすい | 案内不足 | line_sign_confusion |
| M | 汚れ・荒れ | 清掃・維持 | dirty |
| R | 休みにくい | 座る・待つ場所 | area_maintenance_gap |
| S | こわい・不安 | 心理的に安心できない | lonely |
| L | 行き来しにくい | 移動・アクセス | line_step_gap |
| C | 頼れる場がない | 支援・見守り | area_isolation |

### 3.1 被りが大きいペア

| ペア | 被る場面 | 参加者が迷う例 | 整理案 |
|------|----------|----------------|--------|
| **V ↔ S** | 夜の暗い路地で「怖い」 | 「見えないから怖い」vs「雰囲気が怖い」 | **分離を維持しつつ hint を明確化**。V＝視覚・照明、S＝人・雰囲気・一人で不安（暗さ以外） |
| **P ↔ L** | 段差の続く道 | 歩けない vs ルートとして使えない | P＝**その場**の物理障壁、L＝**A地点↔B地点**の移動・アクセス |
| **S ↔ C** | 夜道で一人 | 怖い vs 助けを呼べない | S＝**感情・安心感**、C＝**具体的な支援・見守りの不足** |
| **R ↔ M** | ベンチがない汚い広場 | 休めない vs 荒れている | R＝**機能不足**（座る・待つ）、M＝**状態・清掃** |
| **I ↔ V** | 看板が見えない | 迷う vs 見えない | I＝**情報・案内の不足**、V＝**暗さ・視認性** |

### 3.2 「怖い」と「不安」について

現行 UI では **「こわい・不安」が 1 つの選択肢（S）** にまとまっている。  
別項目として「怖い」「不安」が **2 つ並んでいるわけではない**。

問題は **V（見えにくい）と S（こわい・不安）の境界**：

- 暗くて見えない → **V**
- 明るいが人通りがなく不安 → **S**
- 暗くてかつ不安 → **どちらか一方**（自由記述で補う）

## 4. 減らしてよい／任意にできそうな項目

### 4.1 削減候補（優先度順）

| 項目 | 提案 | 理由 |
|------|------|------|
| **severity（深刻度）** | **任意化 or 削除** | comment と重複しやすい。分析時は文字数・語彙でも代替可 |
| **timeTag** | **デフォルト「常時」＋任意展開** | 多くの投稿は常時で足りる。夜だけ等は comment に書かれる |
| **placeArchetype** | Bad では **6〜7 個に整理** | 「どれにも当てはまらない」は残す。商業・路地などは統合検討可 |
| **needType 8 → 6** | 下記「案 B」参照 | タップ数削減。RQ2 写像は doc 27 の 3 要因軸でカバー可能 |

### 4.2 維持すべき項目

| 項目 | 理由 |
|------|------|
| needType（困りの型） | RQ2 のプラン分岐・分類の要 |
| affectedGroups | 「誰にとって」＝視点置換の核心 |
| comment | 列 C の質を決める。10 字は最低限として妥当 |
| placeArchetype | 場所文脈。RQ2 場所型と接続 |
| 良い / 悪い 2 分類 | 素材の polarity。Good は既に軽い |

### 4.3 足す候補（現状ない）

| 項目 | 必要性 | 備考 |
|------|--------|------|
| **困りの広がり（一点 / 動線 / 周辺）** | RQ2 では重要、RQ1 では **後回し可** | ゲーム側 `BARRIER_SCALE` はあるが AR フォーム未実装。参加者負担増 |
| **投稿者プロフィール（profileTags）** | 低 | Store にフィールドのみ。UI なし。卒論で属性分析するなら **任意 1 問**（例：普段よく歩く人 / 車いす利用 等） |
| **写真内タップ（photoPins）** | 低 | データ構造はある。UI 未実装。段階的導入向き |

## 5. 改訂案

### 案 A — 最小変更（ワークショップ直前向け）

- severity を **任意**（未選択 = mid）
- timeTag を **デフォルト常時**、ステップ 3 からは **「時間帯を指定する（任意）」** に折りたたみ
- needType の hint 文を 3.1 の境界説明に更新（UI のみ、選択肢数は 8 のまま）

**効果:** 実装半日。必須タップ数はほぼ同じだが、スキップ可能項目が増える。

### 案 B — 6 類型に統合（推奨・中程度の改修）

| 新ラベル | 統合元 | needType 案 |
|----------|--------|-------------|
| 歩きにくい | P | P |
| 見えにくい・暗い | V | V |
| 迷いやすい | I | I |
| 汚れ・荒れ・休めない | M + R | M（hint に「ベンチがない」も含む） |
| 不安・こわい | S | S |
| つながり・頼れなさ | L + C | C（hint に「行き来しにくい」も含む） |

**トレードオフ:** L（行き来しにくい）が C に吸収される。バス・フェリー系は comment で補う前提。

**RQ2 写像:** 既存 `NEED_TYPE_TO_DEFAULT_TYPE` を 6 キーに更新すれば足りる。

### 案 C — ステップ統合（Bad を 3 ステップに）

```
1. 種類（良い / 困り）
2. 困り + 場所 + 誰に（1 画面にスクロール）
3. コメント（＋任意: 時間帯・深刻度）
```

**効果:** 体感が「2 画面」に近づく。列 B ◎ に近づく。

## 6. ポイント（pt）とコイン — 混同しない

| | AR アプリ `/ar.html` | ゲーム内 `ARPostingMode` |
|--|----------------------|---------------------------|
| 名称 | **ポイント（pt）** | **コイン** |
| 付与 | `computePoints()` — 10 + ボーナス | `awardPostCoins()` — POST_REWARD 表 |
| 使い道 | **現状なし**（表示・累計のみ） | 種まき・農業など経済 sink |
| 主目的 | 軽いフィードバック（実装途中） | 参加の「味付け」（主動機ではない） |

### 6.1 なぜあるか（設計意図）

[25_AR投稿_参加要因_先行研究レビュー.md](./25_AR投稿_参加要因_先行研究レビュー.md) の結論:

> **継続参加の本丸は「ループの閉じ」（投稿が効いた・直った）**。  
> ポイント・コインは **主動機にすると crowding-out のリスク**。

| 経済 | 位置づけ |
|------|----------|
| **コイン（ゲーム）** | doc 24 — 現地投稿をやや多めに報いる **軽いインセンティブ**。農業ループと共通ウォレット |
| **pt（AR アプリ）** | 図鑑・ホームの **累計表示**。RQ2 には渡らない。研究分析用の「投稿量の可視化」程度 |

### 6.2 RQ1 でどうするか（提案）

| 選択 | 内容 |
|------|------|
| **A. そのまま** | pt は「記録の厚み」に連動（長文 +5 等）。害は小さい |
| **B. 弱める** | 投稿成功トーストだけにして、ホームの pt 表示を削除 |
| **C. 削除** | 卒論で外的報酬を批判するなら、AR 側 pt は **消した方が主張と整合** |

**追加実装は不要。** コインを AR アプリに載せる必要もない（ゲーム経済と分離されている）。

## 9. いいね機能 — 現状と pt 消費案

### 9.1 現状：**いいね機能はない**

| あるもの | ないもの |
|----------|----------|
| ゲーム内「お気に入り」= **建築ブロック**のショートカット保存 | AR 投稿へのいいね / 共感 |
| pt の **獲得と累計表示** のみ | pt の **消費** |
| 図鑑の閲覧・検索・削除 | 他者投稿へのリアクション |

実装箇所: pt は `computePoints()` / `useArPostingStore` のみ。Supabase に `likes` テーブルもなし。

### 9.2 pt → いいね、は筋が良い

[25](./25_AR投稿_参加要因_先行研究レビュー.md) の懸念は「報酬で釣る主動機化」。  
**いいね消費**は次の点で整合しやすい:

| 点 | 内容 |
|----|------|
| 参加の段階 | 投稿（重い）→ いいね（軽い）の **二段階参加** |
| pt の意味 | 貯めるだけでなく **他者の声に使う** = 関係性（SDT） |
| 研究 | 「共感した」ログが RQ2 前の **受容の入口** データになる |

### 9.3 設計案（実装時）

**基本ルール（案）:**

- 自分の投稿にはいいね不可
- **1 pt 消費 = 1 いいね**（所持 pt が足りないと不可）
- 1 投稿あたり 1 人 1 回
- いいねで **投稿者に pt が増える設計は避ける**（インフレ・競争の温床）

**UI 候補:** 記録図鑑・ピンカードに ♡ ＋ 件数

**データ:** Supabase `ar_likes (annotation_id, user_id, created_at)` ＋ RLS

**注意:** いいね数ランキングを前面に出すと SNS 化しやすい。件数表示は **控えめ** が無難。

## 10. 「誰にとって」（affectedGroups）の見直し

### 10.1 現状

6 チップ（複数可・必須）: **みんな / 高齢者 / 車いす / ベビーカー / 子ども / 観光客**

意図（doc 27）: **バリアフリー・まちづくり** の「誰にとっての問題か」。  
属性（男/女・若者）や移動手段（チャリ）まで **網羅する設計ではない**。

### 10.2 足りない例 vs 全部チップ化

| 例 | チップに足す？ | 問題 |
|----|----------------|------|
| 自転車 | △ | 選択肢が際限なく増える |
| 若者 / 高齢以外の年齢 | △ | 年齢区分の設計が別途必要 |
| 男性 / 女性 | × 推奨しない | 夜道の不安等は **状況** で自由記述の方が自然。性別チップは分析も倫理も重い |
| 視覚障害 | ○ 検討可 | 既存 pictogram 系と整合 |
| 妊婦・授乳 | ○ 任意 | ベビーカーと近いが別文脈 |

**結論:** 男/女・若者・チャリまで **固定チップでカバーしようとすると破綻** しやすい。

### 10.3 三つの方針

| 方針 | 内容 | 向いている状況 |
|------|------|----------------|
| **A. 現状維持** | 6 チップ必須のまま | RQ2 の pictogram 表示を最優先 |
| **B. チップ縮小 + 任意テキスト** | 4〜5 チップ（移動・年齢に直結）＋ **「ほか（任意）」1 行** | **推奨**。構造化と自由の両立 |
| **C. チップ廃止 → コメントに統合** | placeholder で誘導 | 摩擦最小。フィルタ・RQ2 UI は弱くなる |

### 10.4 推奨（方針 B）

**必須チップ（4）案:**

| チップ | 理由 |
|--------|------|
| みんな | デフォルト兼ねる |
| 歩行がゆっくりな人 | 高齢・障害・妊婦等を **ラベルで束ね**（「高齢者」単独より包括的） |
| 車いす・ベビーカー | 物理幅・段差と直結 |
| 子ども | 視線・安全の文脈が独特 |

**任意:** テキスト 20 字まで `affectedNote` — 「チャリ」「夜勤の人」等

**「みんな」選択時:** 他チップと排他（現行どおり）か、**みんな＝スキップ相当** にして任意化も可。

### 10.5 自由記述だけに任せるべきか

| | チップのみ | 自由記述のみ | ハイブリッド |
|--|-----------|-------------|-------------|
| 投稿の速さ | 速い | 遅い・ぶれる | 中 |
| 図鑑フィルタ | ◎ | × | ○ |
| RQ2 オーバーレイ | ◎ pictogram | 文章読解依存 | ○ |
| チャリ・性別・夜 | △ | ◎ | ◎（任意欄） |

**自由記述だけ** は RQ1 の列 B（軽さ）と RQ2 の UI の両方で弱い。  
**完全自由に寄せつつ comment に任せる** なら、ステップ 3 から「誰に」を **削除** し、story の placeholder を  
「誰にとって困るか（例：車いす、夜道、チャリ）」に変える手もある。
